# RCA — QR pairing scan triggers a re-lock loop, drops the scan result

- **Symptom:** Tapping "สแกน QR จาก PC dashboard" opens the camera, but on returning to the app the biometric/PIN lock screen appears and demands re-authentication. After unlocking, the app is back on the Feed tab with no pairing applied — repeating the scan hits the same loop every time.

- **Evidence:** `AppRoot` (MainActivity.kt:107-130) sets `unlocked = false` on every `Lifecycle.Event.ON_STOP`, with no exception. Launching the ZXing scanner via `scanLauncher.launch(...)` in `DeviceConnectionScreen` calls `startActivity` under the hood, which sends `MainActivity` through `onStop()` the moment the camera activity covers it — indistinguishable, from `AppRoot`'s point of view, from the user backgrounding the whole app. `unlocked = false` immediately swaps `AppScreen()` out for `LockScreen()`, which unmounts `DeviceConnectionScreen` and, with it, the `rememberLauncherForActivityResult` registration holding the scan-result callback. When the scanner activity finishes and delivers its result, there is no live callback left to receive it — the result is silently dropped. `LaunchedEffect(unlocked)` then force-triggers `authenticate()` because `unlocked` is false, and after re-auth `AppScreen()` remounts from scratch (all `remember` state reset, including navigation position), landing back on Feed with nothing paired.

- **Root Cause:** `AppRoot`'s lock-on-background logic doesn't distinguish "the user backgrounded the app" from "the app itself launched a trusted sub-activity (camera/QR scanner) and is expecting an immediate result back." Both produce the same `ON_STOP` event, but only the first should re-lock.

- **Why it escaped detection:** The lock screen (re-lock-on-`ON_STOP`) and the QR pairing flow (added later, `rememberLauncherForActivityResult` + ZXing) were built independently. Nothing in either code path accounts for the other; there's no test exercising "launch an external activity for a result while unlocked."

- **Prevention:** Added a one-shot `AppLock.suppressNextLock` flag, set immediately before any `startActivity`-triggering call that expects control back in-flow (QR scanner, file-share sheet), and consumed by the `ON_STOP` handler to skip exactly one re-lock. Genuine backgrounding (home button, app switcher, phone lock) is unaffected since nothing sets the flag in those paths.
